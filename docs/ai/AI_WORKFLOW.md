# AI Workflow

Документ о том, как я использовал AI-инструменты в процессе разработки этого проекта.

---

## Инструменты

Основной инструмент — **Claude Code** (Claude Sonnet 4.6) в терминале.
Работал прямо в проекте: AI видел все файлы, мог читать код, предлагать правки и сразу их применять.

Дополнительно — стандартный стек проверок:

| Инструмент | Для чего |
|---|---|
| ESLint 10 | Статический анализ после каждого изменения |
| Prettier | Форматирование — чтобы не думать о стиле |
| vue-tsc | TypeScript-проверка шаблонов Vue |
| Vitest | Быстрая верификация логики через тесты |
| Playwright | Проверка реальных пользовательских сценариев |

## Как строил итерации

Я не использовал AI для написания проекта с нуля. Архитектура, бизнес-логика и продуктовые решения — всё это я делал сам. AI подключался точечно, по конкретным задачам.

Цикл выглядел так:

```
Я формулирую задачу
        │
        ▼
  AI предлагает решение или находит проблемы
        │
        ▼
  Запускаю проверки: ESLint + TypeScript + тесты
        │
        ├── Есть ошибки → уточняю промпт → повтор
        │
        └── Всё чисто → проверяю вручную в браузере
                              │
                              ├── Есть замечания → правлю
                              └── Всё ок → следующая задача
```

Большинство задач закрывалось за 1–2 итерации. Дольше всего шли задачи на дизайн — там субъективно, приходилось уточнять.

## Для чего использовал AI

### Аудит кода
Самое полезное применение. Просил AI проверить проект целиком по чеклисту: HTML-семантика, доступность, XSS-уязвимости, FSD-правила импортов, мёртвый код. AI находил то, что я пропустил или отложил: отсутствие focus trap в модалке, `v-html` с SVG в AdminLayout, несвязанные `<label>` с `<input>`.

### Генерация инфраструктуры
ESLint flat config, Vitest, Playwright, GitHub Actions — всё это настраивал через AI. Описывал требования, получал готовый конфиг, проверял что он работает. Быстрее, чем читать документацию с нуля.

### Написание тестов
Давал AI функцию или компонент, просил покрыть все пути выполнения. Проверял что тесты реально тестируют логику, а не просто зелёные.

### Разовые рефакторинги
Когда нужно было применить один паттерн к нескольким файлам сразу — например, добавить `useId()` в AppInput, AppSelect и AppTextarea — AI делал это за один раз без копипасты.

### Документация
README, описание архитектуры — AI хорошо структурирует текст. Я давал структуру и ключевые моменты, AI оформлял в читаемый документ.

## Ключевые инженерные решения

### Feature-Sliced Design
Выбрал FSD с самого начала — при росте проекта структура по типам файлов превращается в свалку. FSD даёт однонаправленные импорты и изоляцию слайсов: удалить или добавить фичу можно не боясь сломать другое. AI помогал проверять что правила импортов не нарушаются.

### Cursor-based pagination
Offset-пагинация ломается когда кто-то добавляет запись пока ты листаешь — видишь дубликаты или пропуски. Курсорная пагинация этой проблемы не имеет. Реализовал через `useCursorList` — универсальный composable, который подключается к любому списку.

### IntersectionObserver вместо scroll event
`scroll` срабатывает сотни раз в секунду. `IntersectionObserver` — только когда нужный элемент появляется в viewport. Меньше нагрузки, чище код, работает из коробки везде.

### useId() из Vue 3.5
Нужно было связать `<label>` с `<input>` через уникальные ID. Первый вариант — модульный счётчик — ESLint отклонил. Vue 3.5 решает это встроенным `useId()`: стабильный ID на каждый экземпляр компонента, SSR-совместимый, без побочных эффектов.

### ESLint flat config
ESLint 10 переходит на flat config как стандарт. Начинать с legacy-формата — значит тащить технический долг с первого дня. Настроил сразу правильно, один раз.

## Что AI не делал

- Не проектировал архитектуру — это я решал сам
- Не принимал продуктовые решения
- Не писал бизнес-логику: авторизацию, управление офферами, фильтрацию товаров
- Не выбирал стек — Vue 3, Pinia, Vue Router, Axios были определены до начала работы с AI

AI — инструмент, который ускоряет конкретные задачи. Не замена разработчику.
